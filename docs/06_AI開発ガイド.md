# AI 開発ガイド（初心者向け）

このプロジェクトは **Claude Code などの AI コーディングエージェントと一緒に開発する**前提で整備されています。
このガイドは「クローンした直後から、AI に指示して安全に開発を進める」までの手順を、AI 開発が初めてのメンバー向けにまとめたものです。

---

## 1. 事前準備（クローン〜セットアップ）

まず通常のセットアップを済ませます（README のクイックスタートと同じ）。

```bash
git clone <このリポジトリのURL>
cd oki-dive-partner

# Windows (PowerShell)
powershell -ExecutionPolicy Bypass -File scripts\setup.ps1
# Mac / Linux / WSL / Git Bash
bash scripts/setup.sh

npm run dev   # → http://localhost:3000 が開けば準備OK
```

ここまでで「AI がいなくても動く状態」を必ず確認してください。
**AI に作業させる前に動く状態を作っておく**のが鉄則です（壊れたときに AI のせいか元からか切り分けられるため）。

---

## 2. AI ツール（Claude Code）の導入

### インストール

- **デスクトップアプリ / VS Code 拡張**: https://claude.com/claude-code からインストール
- **CLI（ターミナル派向け）**:
  ```bash
  npm install -g @anthropic-ai/claude-code
  ```

### プロジェクトで起動する

**必ずプロジェクトのルート（`oki-dive-partner/` 直下）で起動してください。**

```bash
cd oki-dive-partner
claude
```

起動場所が重要な理由：

1. ルートにある **`CLAUDE.md` が自動で読み込まれ**、AI が技術スタック・開発規約・「やってはいけないこと」を最初から知っている状態になる
2. `docs/` の設計資料も AI がすぐ参照できる
3. `.claude/launch.json` により、AI が開発サーバーを起動して動作確認までできる

> つまり **このリポジトリをクローンした時点で、AI への「プロジェクト設定」はほぼ完了しています。** 個人で追加設定は不要です。

### 初回に許可を求められたら

AI がファイル編集やコマンド実行のたびに許可を求めてきます。最初のうちは：

- `npm run dev` / `npm run typecheck` / `git status` などの**読み取り・確認系 → 許可してOK**
- `git push` / ファイル削除 / `.env.local` を触る操作 → **内容を読んでから判断**

慣れてきたら「今後は常に許可」を選んで手間を減らせます。

## 3. 作業の進め方（1タスク = 1ブランチ）

AI 開発でもチーム開発の基本は同じです。**main に直接コミットせず、タスクごとにブランチを切ります。**

### 手順（毎回これ）

```bash
# ① 最新の main を取り込む
git checkout main
git pull

# ② 作業ブランチを作る（名前は feature/内容 で）
git checkout -b feature/customer-search

# ③ Claude Code を起動して指示する
claude
```

ブランチ作成自体を AI に頼んでも構いません：

> 「main から feature/customer-search ブランチを切って、顧客一覧に氏名検索を追加して」

### 並行して複数タスクをやりたいとき（発展）

ブランチを切り替えると作業中のファイルが混ざって混乱しがちです。
**git worktree** を使うと、ブランチごとに別フォルダを作って同時に開発できます：

```bash
# 隣に別フォルダとして feature ブランチの作業場を作る
git worktree add ../oki-dive-partner-search feature/customer-search
cd ../oki-dive-partner-search
npm install   # フォルダごとに node_modules が必要
claude
```

不要になったら `git worktree remove ../oki-dive-partner-search` で片付けます。
※ 最初のうちは worktree を使わず「1度に1タスク」で十分です。

## 4. AI への指示のコツ

### 良い指示の型

**「どこの資料に基づいて」「何を」「完了条件は何か」** を伝えると精度が上がります。

| ❌ 曖昧 | ✅ 具体的 |
|---|---|
| 検索機能つけて | docs/01_要件定義書.md の顧客台帳機能 No.1 にある「氏名・最終来店日で検索」を顧客一覧画面に実装して。typecheck が通ることを確認して |
| バグ直して | 予約一覧で日付を変えても表示が更新されない。原因を調査して報告して。修正はまだしないで |
| きれいにして | src/lib/api.ts の重複した fetch 処理を共通化して。動作は変えないこと |

### このプロジェクトならではの伝え方

- 仕様の根拠は `docs/` にあります。「**要件定義書の◯◯を見て**」「**DB設計編のシート定義に合わせて**」と資料名で指示すると、AI が勝手な仕様を発明しません
- 画面の変更は「**npm run dev で実際に画面を開いて確認して**」まで指示する
- データ構造を変えるときは「**docs/03_基本設計書_DB設計編.md と矛盾しないか確認して**」を付ける

### 大きめのタスクは「計画 → 実装」の2段階で

いきなり書かせず、まず計画を出させます：

> 「問診票のQRコード読取機能を実装したい。まず docs/04_詳細設計書.md を読んで実装計画を出して。コードはまだ書かないで」

計画に納得してから「その計画で進めて」と指示します。手戻りが激減します。

## 5. AI が書いたコードの受け入れチェック

**AI の出力をそのままマージしない**こと。最低限これを回してください：

```bash
npm run typecheck   # 型チェック（必ず通す）
npm run lint        # ESLint
npm run dev         # 実際に画面を開いて、変更したフローを自分で操作する
```

チェックの観点：

1. **動くか** — 変更した画面・APIを実際に触る（Sheets 未設定のモックモードで大半確認できます）
2. **仕様どおりか** — `docs/` の該当箇所と見比べる
3. **やりすぎていないか** — 頼んでいないファイルまで変更されていないか `git diff` で確認
4. **秘密情報が混ざっていないか** — `.env.local` の値やサービスアカウント鍵がコードに直書きされていないか

AI 自身にレビューさせるのも有効です：

> 「今回の変更を CLAUDE.md の規約と docs/01_要件定義書.md のセキュリティ要求に照らしてセルフレビューして」

問題なければコミット → push → Pull Request を作り、**人間のレビューを1人以上**通してから main にマージします。

## 6. やってはいけないこと（重要）

| 禁止事項 | 理由 |
|---|---|
| `.env.local` の中身を AI とのチャットに貼る | パスワード・秘密鍵が学習外でもログに残るリスク。「.env.local を参照して」とファイル名で伝えれば十分 |
| サービスアカウント鍵（JSON）をリポジトリに置く | 顧客の個人情報・健康情報への全権アクセス鍵。漏洩したら即失効・再発行 |
| AI の出力を確認せず main に push | このアプリは健康情報を扱う。壊れた検証ロジックは事故に直結する |
| 「全部いい感じにして」で長時間放置 | 意図しない大規模変更が起きる。タスクは小さく区切る |
| 本番データ（実際の顧客情報）でテスト | 開発はモックデータ（`src/lib/mockData.ts`）で行う |

---

## 7. 困ったときは

- **AI が変な方向に進み始めた** → 遠慮なく Esc で止めて指示し直す。「さっきの変更を全部取り消して」も有効（git があるので戻せます）
- **変更がぐちゃぐちゃになった** → `git checkout .` で最後のコミットまで戻す（コミット前の変更が全部消えるので注意）
- **エラーの意味がわからない** → エラーメッセージをそのまま AI に貼って「原因を説明して。まず修正はしないで」
- **仕様がわからない** → AI に「docs/ から◯◯の仕様を探して要約して」と聞く

## 付録: 最初の1タスクにおすすめの練習

クローン直後の肩慣らしとして、この順で試すのがおすすめです：

1. 「このプロジェクトの構成を説明して」（読むだけ・安全）
2. 「ダッシュボードの見出しの文言を◯◯に変えて、画面で確認して」（小さな変更＋動作確認）
3. 「docs/01_要件定義書.md の機能一覧のうち、未実装のものがあるか調べて」（資料とコードの突き合わせ）

ここまでできれば、実タスクに入れます。
