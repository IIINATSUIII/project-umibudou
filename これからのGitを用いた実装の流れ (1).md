# Git チーム開発ガイド｜沖ダイブ・パートナー

## ブランチ戦略

**GitHub Flow**（シンプルで小規模チームに向いてる）を採用。

```
main          ← 常に動く状態を保つ（本番 = Firebase Hosting）
  └── feature/auth          ← 認証機能
  └── feature/reservation   ← 予約機能
  └── feature/questionnaire ← 問診票機能
  └── feature/customer      ← 顧客台帳機能
  └── fix/qr-bug            ← バグ修正
```

ルールはシンプルに2つだけ：

- `main` に直接 push しない
- 機能・修正ごとにブランチを切る → PR → レビュー → マージ

| 担当 | 担当機能 |
|---|---|
| Aさん | 環境構築・認証・Sheets基盤（全員の土台を先に作る） |
| Bさん | 予約集約機能 |
| Cさん | 電子問診票機能（工数大きいので1〜2人） |
| Dさん | 〃（問診票のQR部分など） |
| Eさん | 顧客台帳機能 |
| Fさん | UI全体・レスポンシブ・テスト・デプロイ |

## 実際の進め方（1サイクル）

```
1. GitHub Issues でタスクを作る
   例：「予約CSVインポート機能を作る」

2. ブランチを切る
   git checkout -b feature/csv-import

3. VS Code で実装（Coworkに書かせながら）

4. コミット（細かくこまめに）
   git commit -m "feat: CSVパース関数を追加"

5. GitHubへ push → Pull Request を作る

6. チームの誰か1人がレビュー → 承認 → main にマージ
```

---

## 最低限決めておくこと

### コミットメッセージのルール

揃えないと履歴が読めなくなる。

```
feat:  新機能追加
fix:   バグ修正
chore: 設定・環境変更
docs:  ドキュメント
```

### PRのルール

- 1PR = 1機能 or 1修正（大きくしすぎない）
- レビュアーを必ず1人つける
- `main` へのマージは承認後のみ

### .envファイルは絶対にGitに入れない

```gitignore
# .gitignore に必ず追加
.env.local
```

Firebase秘密鍵・スプレッドシートIDなどは各自がローカルで持つ。本番用は Firebase の環境変数設定で管理。

## Coworkとの組み合わせ方

各メンバーが自分のブランチでCoworkに「この機能を実装して」と頼む → コードが出てくる → 自分でテスト → PRを出す、という流れが自然。

Coworkが書いたコードをそのままコミットするのではなく、**自分がレビューして理解してからコミットする**習慣をつけると、後でバグが出たときに詰まらなくて済む。
