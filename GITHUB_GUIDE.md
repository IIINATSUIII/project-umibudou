# GitHub 初心者ガイド（project-umibudou チーム向け）

このドキュメントは、GitHubを使い始めたばかりのメンバーでも迷わず開発に参加できるようにするための手引きです。
わからなくなったら、まずこのページに戻ってきてください。

---

## 1. 事前準備

### 1-1. Gitをインストールする
- Windows: [Git for Windows](https://gitforwindows.org/) をダウンロードしてインストール
- インストール後、コマンドプロンプトやPowerShellで以下を実行して確認

```bash
git --version
```

バージョンが表示されればOKです。

### 1-2. GitHubアカウントを作る
[https://github.com](https://github.com) でアカウントを作成してください。

### 1-3. Gitに自分の名前とメールアドレスを設定する
（コミットした人が誰かわかるようにするための設定です）

```bash
git config --global user.name "あなたの名前"
git config --global user.email "あなたのメールアドレス"
```

### 1-4. リポジトリにアクセスできるようにする
チームのオーナー（管理者）から、このリポジトリの **Collaborator（共同作業者）** として招待してもらってください。
GitHubから届く招待メールを承認すると準備完了です。

## 2. 最初に覚える用語集

| 用語 | 意味 |
|---|---|
| リポジトリ（repo） | プロジェクトの保管庫。ファイルと変更履歴が入っている場所 |
| クローン（clone） | リポジトリを自分のPCにコピーすること（最初の1回だけ行う） |
| コミット（commit） | 変更内容を記録する「セーブポイント」 |
| ブランチ（branch） | mainとは別の作業スペース。ここで自分の作業を進める |
| プッシュ（push） | 自分のPCの変更をGitHub上に反映すること |
| プル（pull） | GitHub上の最新の変更を自分のPCに取り込むこと |
| プルリクエスト（PR） | 「この変更をmainに取り込んでください」という提案・依頼 |
| マージ（merge） | PRの内容をmainブランチに取り込むこと |
| コンフリクト（conflict） | 複数人が同じ箇所を変更してGitが自動で統合できない状態 |

---

## 3. 最初の1回だけ行うこと（クローン）

```bash
git clone https://github.com/<オーナー名>/project-umibudou.git
cd project-umibudou
```

これでプロジェクトのファイル一式が自分のPCにコピーされます。

---

## 4. 基本の作業の流れ（毎回これを繰り返す）

**ポイント: mainブランチには直接作業せず、必ず自分専用のブランチを作って作業します。**

### Step 1. mainを最新の状態にする

```bash
git checkout main
git pull origin main
```

### Step 2. 作業用のブランチを作る

```bash
git checkout -b feature/ログイン画面作成
```

ブランチ名の付け方は [6章](#6-ブランチ名の付け方) を参照してください。

### Step 3. ファイルを編集する

普段通りコードを書いたり、ファイルを変更したりします。

### Step 4. 変更を確認する

```bash
git status
```

変更したファイルの一覧が表示されます。

### Step 5. 変更をコミットする

```bash
git add .
git commit -m "ログイン画面のレイアウトを作成"
```

- `git add .` : 変更したファイルをすべて「コミット対象」にする
- `git commit -m "メッセージ"` : メッセージを付けてセーブする

コミットメッセージの書き方は [7章](#7-コミットメッセージの書き方) を参照してください。

### Step 6. GitHubにプッシュする

```bash
git push origin feature/ログイン画面作成
```

（初回プッシュ時は `git push -u origin ブランチ名` としても構いません）

### Step 7. プルリクエスト（PR）を作成する

1. GitHubのリポジトリページを開く
2. 「Compare & pull request」ボタンをクリック
3. タイトルと説明を書く（何を・なぜ変更したか）
4. 「Create pull request」をクリック
5. チームメンバーにレビューを依頼する（Slackなどで一声かけると親切です）

### Step 8. レビューしてもらう・マージする

- レビュアーがコメントを付けたら、修正してまた `git add` → `git commit` → `git push` すればPRに自動で反映されます
- OKが出たらGitHub上の「Merge pull request」ボタンでmainに取り込みます
- マージ後、不要になった作業ブランチは削除してOKです（GitHub上に削除ボタンがあります）

### Step 9. 自分のPCのmainも更新する

```bash
git checkout main
git pull origin main
```

これで次の作業に進めます。

---

## 5. まとめ図

```
main を最新化
   ↓
作業ブランチを作成
   ↓
編集 → add → commit（何度でもOK）
   ↓
push
   ↓
プルリクエスト作成
   ↓
レビュー・修正
   ↓
マージ（mainに反映）
   ↓
最初に戻る
```

## 6. ブランチ名の付け方

わかりやすさのため、以下のルールを推奨します。

```
種別/内容の概要
```

| 種別 | 用途 | 例 |
|---|---|---|
| `feature/` | 新機能追加 | `feature/login-page` |
| `fix/` | バグ修正 | `fix/button-color` |
| `docs/` | ドキュメント修正 | `docs/readme-update` |

---

## 7. コミットメッセージの書き方

**「何をしたか」が一目でわかるように短く書きます。**

良い例:
- `ログイン画面のUIを追加`
- `APIエラー時のメッセージを修正`

避けたい例:
- `更新`
- `いろいろ直した`
- `wip`（作業中で一旦保存したい場合を除く）

---

## 8. コンフリクト（衝突）が起きたら

複数人が同じファイルの同じ箇所を変更すると、マージ時にコンフリクトが起きることがあります。焦らなくて大丈夫です。

1. `git pull origin main` などをした際に以下のような表示が出たらコンフリクト発生のサインです

```
CONFLICT (content): Merge conflict in ファイル名
```

2. 該当ファイルを開くと、以下のような印が入っています

```
<<<<<<< HEAD
自分の変更内容
=======
相手の変更内容
>>>>>>> main
```

3. どちらを残すか（あるいは両方活かすか）を話し合って決め、`<<<<<<<`、`=======`、`>>>>>>>` の行を削除してファイルを正しい内容に整えます
4. 保存したら通常通りコミットします

```bash
git add .
git commit -m "コンフリクトを解消"
```

**わからなくなったら、一人で悩まず必ずチームに相談してください。**

## 9. よくあるトラブルと対処法

| 状況 | 対処法 |
|---|---|
| `git push` が失敗する（rejected） | 先に `git pull origin <ブランチ名>` してから再度pushする |
| 間違えてmainに直接コミットしてしまった | すぐにチームに共有し、一緒に対処する（自己判断でresetやforce pushはしない） |
| コミットメッセージを打ち忘れた/直したい | 直後であれば `git commit --amend` で修正可能（ただし既にpush済みなら要相談） |
| ファイルを間違えてステージしてしまった | `git restore --staged ファイル名` で取り消せる |

---

## 10. チームのお約束

- mainブランチに直接pushしない（必ずPR経由）
- 作業を始める前に必ず `git pull origin main` で最新化する
- PRは最低1人のレビューを経てからマージする
- 詰まったら早めにチームに相談する（ひとりで抱え込まない）
